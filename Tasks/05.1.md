## CONTEXTO

Esta task conecta cliente e profissional em tempo real. O motor de matching usa PostGIS para buscar profissionais proximos e BullMQ para gerenciar a fila. Socket.IO notifica profissionais em tempo real. E o coracao do MechaGo.

## BACKEND

### 1. Modulo Matching (BullMQ + PostGIS)

**Referencia**: Technical Reference secao 3.1 (matching/)

Criar `apps/api/src/modules/matching/`:

```
matching/
├── matching.service.ts       # Logica de busca PostGIS + orquestracao
├── matching.queue.ts         # Definicao da fila BullMQ
├── matching.worker.ts        # Worker que processa jobs
└── matching.service.test.ts  # Testes TDD
```

**Query PostGIS** (conforme Technical Reference secao 4 e RULES.md secao 4):

```sql
SELECT p.*, u.name, u.rating,
  ST_Distance(
    ST_SetSRID(ST_MakePoint(p.longitude, p.latitude), 4326)::geography,
    ST_SetSRID(ST_MakePoint($lng, $lat), 4326)::geography
  ) AS distance_meters
FROM professionals p
JOIN users u ON p.user_id = u.id
WHERE p.is_online = true
  AND ST_DWithin(
    ST_SetSRID(ST_MakePoint(p.longitude, p.latitude), 4326)::geography,
    ST_SetSRID(ST_MakePoint($lng, $lat), 4326)::geography,
    $radiusMeters
  )
  AND $vehicleType = ANY(p.vehicle_types_served)
ORDER BY u.rating DESC, distance_meters ASC
LIMIT 10
```

**Fluxo BullMQ**:

1. Cliente cria service-request (status: `pending`)
2. Service muda status para `matching` e adiciona job na fila
3. Worker executa query PostGIS
4. Se encontrou profissionais: envia `new_request` via Socket.IO + FCM push
5. Timeout 3min: se ninguem aceitou, muda para `waiting_queue`
6. Fila de espera: informa posicao ao cliente via Socket.IO

### 2. Socket.IO Setup

Criar `apps/api/src/socket.ts`:

- Integrar Socket.IO com Hono (mesmo server HTTP)
- Auth via JWT no handshake
- Rooms: `request:{requestId}`, `professional:{professionalId}`
- Events conforme Technical Reference secao 6.8:

```
Server → Client:
  new_request       # profissional recebe chamado
  status_update     # cliente recebe mudanca de status
  queue_update      # cliente na fila recebe posicao

Client → Server:
  join_request      # entrar na room do request
```

### 3. Notifications (FCM basico)

Criar `apps/api/src/modules/notifications/notifications.service.ts`:

- Enviar push via Firebase Admin SDK
- Por enquanto: notificar profissional quando ha novo chamado
- Payload: { title, body, data: { requestId, type } }

### 4. Endpoints de Service-Requests (complementar baseline atual de service-requests)

Adicionar aos endpoints existentes:

```
POST /api/v1/service-requests/:id/accept    # Profissional aceita
POST /api/v1/service-requests/:id/arrived   # Profissional chegou (<200m GPS)
```

**accept**: muda status `matching` → `accepted`, seta professionalId, notifica cliente
**arrived**: valida GPS (ST_DWithin 200m), muda `professional_enroute` → `professional_arrived`
