# Task 05 — Matching Engine & App Pro (Onboarding)

> Sprint: 4 | Pré-requisito: Task 04 (Service Requests)
> Nível: Arquiteto Sênior / Real-time Systems

## CONTEXTO

Nesta task, o MechaGo ganha vida. Implementaremos o motor que conecta o pedido do cliente ao profissional mais próximo e iniciaremos o desenvolvimento do aplicativo **MechaGo Pro**.

## MARCOS DE CONCLUSÃO (BACKEND)

### 1. Motor de Matching (PostGIS + BullMQ)
- [ ] Criar `apps/api/src/modules/matching/matching.service.ts`.
- [ ] Implementar a query de busca geoespacial:
  - Filtrar profissionais: `isOnline: true`, `vehicleTypesServed` inclui o do cliente, dentro do `radiusKm`.
  - Ordenar por: `rating` DESC e `distância` ASC.
- [ ] Configurar fila `matching-queue` no BullMQ para gerenciar as tentativas de oferta.
- [ ] Implementar lógica de "Fila de Espera" se nenhum profissional for encontrado/aceitar em 3 minutos.

### 2. Notificações & Real-time
- [ ] Implementar `notifications.service.ts` usando Firebase Admin (FCM).
- [ ] Criar evento Socket.IO `new_request` para alertar profissionais em tempo real.

## REGRAS DE OURO PARA ESTA TASK

1. **Simetria**: O App Pro deve seguir rigorosamente a mesma arquitetura de pastas e padrões de código do App Cliente.
2. **PostGIS**: A busca de profissionais DEVE usar `ST_DWithin` para performance.
3. **Fidelidade Stitch**: O App Pro possui um visual "Noir" levemente diferente do Cliente (mais focado em dados). Seguir os arquivos da pasta `DesignPro/`.
