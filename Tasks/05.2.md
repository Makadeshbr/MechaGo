# Task 05.2 — Tracking + Diagnostico + Resolucao

> Sprint: 4 | Pre-requisito: Task 05.1 (Matching Engine)
> Fatia vertical: Profissional navega ate o cliente → chega → diagnostica → resolve ou escala

## CONTEXTO

Com o matching funcionando (Task 05.1), agora implementamos o fluxo de atendimento completo: o profissional navega ate o cliente com GPS em tempo real, chega, faz o diagnostico com foto obrigatoria, e resolve o problema (ou escala para guincho). Esta task cobre os estados 4-9 da maquina de estados.

## BACKEND

### 1. Modulo Tracking (Socket.IO rooms + GPS)

**Referencia**: Technical Reference secao 3.1 (tracking/)

Criar `apps/api/src/modules/tracking/`:

```
tracking/
├── tracking.gateway.ts      # Socket.IO event handlers
└── tracking.service.ts      # Logica de tracking
```

**Socket.IO Events** (Technical Reference secao 6.8):

```
Client → Server:
  join_request        # Cliente/profissional entram na room
  update_location     # Profissional envia GPS periodicamente

Server → Client:
  professional_location  # Cliente recebe posicao + ETA
  status_update          # Ambos recebem mudancas de status
```

**Room pattern**: `request:{requestId}` — ambos (cliente e profissional) entram na room.

### 2. Modulo Uploads (presigned URL)

Criar `apps/api/src/modules/uploads/`:

```
uploads/
├── uploads.routes.ts       # POST /uploads/presigned-url
├── uploads.service.ts      # Gerar presigned URL (Cloudflare R2)
└── uploads.schemas.ts      # Zod schema
```

**Endpoint**: `POST /api/v1/uploads/presigned-url`

- Request: `{ fileName, contentType, context: "diagnosis" | "completion" | "avatar" }`
- Response: `{ uploadUrl, publicUrl, expiresIn }`
- Usar AWS S3 SDK com endpoint R2 (S3-compatible)

**Alternativa MVP**: Se R2 nao estiver configurado, usar upload direto para o server com multer/formidable e salvar em disco local. Perguntar ao desenvolvedor.

### 3. Endpoints de Service-Requests (complementar)

Adicionar:

```
POST /api/v1/service-requests/:id/diagnosis
  Body: { diagnosisNotes, diagnosisPhotoUrl, canResolveOnSite }
  Regra: foto OBRIGATORIA (rejeitar sem diagnosisPhotoUrl)
  Status: professional_arrived → diagnosing

POST /api/v1/service-requests/:id/resolve
  Body: { finalPrice, completionPhotoUrl, priceJustification? }
  Regras:
    - Foto OBRIGATORIA (rejeitar sem completionPhotoUrl)
    - Se finalPrice fora de ±25% da estimativa: priceJustification OBRIGATORIA
    - Calcula priceDeviation = |finalPrice - estimatedPrice| / estimatedPrice
  Status: diagnosing → resolved

POST /api/v1/service-requests/:id/escalate
  Body: { escalationReason, needsTow }
  Status: diagnosing → escalated
  Se needsTow: status → tow_requested (guincho e V1.5, por ora so muda status)

POST /api/v1/service-requests/:id/approve-price
  Status: resolved → completed (se pre-pago) ou → aguardando pagamento
  Regra: so o cliente pode aprovar

POST /api/v1/service-requests/:id/contest-price
  Body: { contestReason }
  Regra: gera disputa para admin resolver (MVP: log + notificacao)

POST /api/v1/service-requests/:id/complete
  Status: → completed
  Regra: verificar que foto de conclusao existe
```
