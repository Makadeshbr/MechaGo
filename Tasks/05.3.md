## BACKEND

### 1. Modulo Payments (Mercado Pago Pix + Split)

**Referencia**: Technical Reference secao 3.1 (payments/)

Criar `apps/api/src/modules/payments/`:

```
payments/
├── payments.schemas.ts      # Zod schemas
├── payments.routes.ts       # Rotas
├── payments.service.ts      # Logica de pagamento
├── payments.repository.ts   # Queries Drizzle
├── payments.webhook.ts      # Webhook handler + HMAC validation
└── payments.service.test.ts # Testes TDD
```

**Endpoints** (Technical Reference secao 6.5):

```
POST /api/v1/payments/create-diagnostic   # Pre-pagamento taxa diagnostico (30% da estimativa)
POST /api/v1/payments/create-service      # Pagamento final do servico
POST /api/v1/payments/webhook/mercadopago # Webhook (SEM auth JWT — usa HMAC)
GET  /api/v1/payments/:id                 # Status do pagamento
```

**Fluxo Pix**:

1. Cliente aprova estimativa → `create-diagnostic` gera Pix (30%)
2. Mercado Pago processa → webhook confirma → status: `paid`
3. Apos servico resolvido + preco aprovado → `create-service` gera Pix (valor final - diagnostico)
4. Split automatico: profissional recebe % liquido, plataforma recebe comissao
5. Comissao MVP: 0% (fundador), depois 10%

**Webhook HMAC** (RULES.md secao 3 — seguranca):

- Validar assinatura HMAC-SHA256 do Mercado Pago
- Rejeitar webhooks com assinatura invalida
- Logar tentativas invalidas (possivel ataque)

### 2. Modulo Reviews (bilateral com tags)

Criar `apps/api/src/modules/reviews/`:

```
reviews/
├── reviews.schemas.ts
├── reviews.routes.ts
├── reviews.service.ts
├── reviews.repository.ts
└── reviews.service.test.ts
```

**Endpoints** (Technical Reference secao 6.6):

```
POST /api/v1/reviews                      # Criar avaliacao (cliente ou profissional)
GET  /api/v1/reviews/professional/:id     # Avaliacoes de um profissional
```

**Regras**:

- Avaliacao bilateral: cliente avalia profissional E profissional avalia cliente
- Rating: 1-5 estrelas
- Tags predefinidas: "pontual", "educado", "competente", "justo", "rapido" (cliente → pro); "local_acessivel", "comunicativo", "pagou_rapido" (pro → cliente)
- So pode avaliar apos status `completed`
- Nota minima ativa do profissional: 3.5 (abaixo = suspensao temporaria)
- Uma avaliacao por service_request por reviewer

### 3. Cancelamento (6 cenarios — INEGOCIAVEL)

Implementar em `service-requests.service.ts`:

```
PATCH /api/v1/service-requests/:id/cancel
  Body: { reason?, cancelledBy: "client" | "professional" }
```

**Os 6 cenarios** (conforme Technical Reference):

```typescript
// Cenario 1: Cliente cancela em ate 2 min → 100% reembolso
// Cenario 2: Cliente cancela apos 2 min (prof ja aceitou) → 70% reembolso
// Cenario 3: Cliente cancela com prof a caminho → 0% reembolso
// Cenario 4: Profissional cancela → auto-rematch + penalidade (max 2/mes, 3o = 48h suspensao)
// Cenario 5: Profissional nao chega no SLA → 100% reembolso + penalidade pro
// Cenario 6: Ninguem aceita em 5 min → fila de espera, cobranca so no accept
```
