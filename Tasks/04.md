# Task 04 — Service Requests & Pricing Engine (V3)

> Sprint: 3 | Pré-requisito: Task 03 (Users + Vehicles)
> Nível: Sênior / Enterprise Architect

## CONTEXTO

Esta task implementa o "Core" do MechaGo: a capacidade de um cliente solicitar socorro com um preço calculado dinamicamente. O sistema deve detectar automaticamente se o usuário está em rodovia ou cidade e aplicar os multiplicadores da **Fórmula V3**.

## MARCOS DE CONCLUSÃO (BACKEND)

### 1. Pricing Engine (TDD)
- [ ] Criar `apps/api/src/modules/service-requests/pricing.service.ts`.
- [ ] Implementar `calculateEstimate` com os 5 multiplicadores:
  - **Base**: Média da `price_tables` por `problemType` + `vehicleType`.
  - **Veículo**: car(1.0), moto(0.85), suv(1.15), truck(2.0).
  - **Horário**: Dia(1.0), Noite(22h-6h)(1.25), Feriado(1.20).
  - **Local**: Urbano(1.0), Rodovia(1.15), Rural(1.30).
  - **Distância**: ≤5km(1.0), 5-15km(1.10), 15km+(1.25).
- [ ] **Testes (Obrigatório)**: Criar `pricing.service.test.ts` cobrindo pelo menos 10 combinações de preços.

### 2. Geofencing (PostGIS)
- [ ] Implementar no `ServiceRequestsRepository` a busca na tabela `roadway_info`.
- [ ] Usar query PostGIS para verificar se `ST_Contains` ou `ST_DWithin` do ponto do cliente em relação aos polígonos de rodovias.
- [ ] Retornar o `context` (urban/highway) e o telefone da concessionária se houver.

### 3. Módulo de Service-Requests
- [ ] `service-requests.schemas.ts`: Definir inputs/outputs usando os tipos da `shared`.
- [ ] `service-requests.repository.ts`: Criar pedido com status `pending`.
- [ ] `service-requests.service.ts`: Orquestrar Geofencing + Pricing + Repository.
- [ ] `service-requests.routes.ts`: Expor `POST /service-requests` e `GET /service-requests/:id`.
